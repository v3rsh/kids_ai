какие сценарии тестировать. Без этого риск, что задача потеряется

Привязка к ТЗ — обязательна (`§…` или ссылка на пункт), чтобы при

## WAVE4-LINKS-UX — пользовательский UX режима LINKS (§33.6)

**Контекст.** В БД режим приёма `intake_mode` уже корректно прокидывается
из `services.intake_mode.get_intake_mode()` в `create_application` (Wave 4
A2). Реестр (§25.1 поле №13, §33.6.3) различает FILES и LINKS через
helper `view_command_or_link` в `app/services/registry.py`. Что осталось
— это пользовательский поток: при `intake_mode = LINKS` бот должен
запрашивать ссылку на облачную папку вместо файла.

**Текущее поведение.**

- `app/handlers/user_files.py` всегда работает в FILES-режиме: спрашивает
  файл вложением, валидирует расширение/размер (§16, §11.4), кладёт во
  временный каталог.
- `app/handlers/user_confirm.py` материализует файлы через
  `services.storage.rename_and_save_file` независимо от режима.
- При `intake_mode = LINKS` (после ручного `/intake_mode links` или
  авто-переключения по диску §28.1) пользователь всё равно видит UI
  «загрузите файл», а его заявка попадёт в БД с пустым `cloud_link` —
  то есть в реестре поле №13 будет пустым.

**Что нужно сделать.**

1. **`app/handlers/user_files.py`** — после установки FSM-state
   `user_intake_files_collect` спросить `get_intake_mode()`:
   - если FILES — старое поведение (приём файла вложением);
   - если LINKS — показать новый prompt «пришлите ссылку на облачную
     папку» и валидировать URL (https/http, длина, базовая sanity).
2. **Новый FSM-state** для приёма ссылки в `app/states.py`
   (например, `UserIntake.user_intake_link_collect`) — чтобы не
   смешивать парсер ссылки с приёмом файла.
3. **`app/handlers/user_confirm.py` → `cmd_submit`**:
   - в LINKS-режиме передавать `cloud_link=<введённый URL>` в
     `applications_service.create_application` (поле уже есть в API).
   - пропускать материализацию файлов (`_materialize_files` не
     вызывается для LINKS) — диск не нужен.
4. **Тексты §13 / §14 / §18**: формулировки про «файлы» заменить на
   «ссылку», когда режим LINKS, чтобы UI не путал родителя.
5. **Тесты:**
   - `tests/test_application_flow.py` — параметризованный кейс: подача
     заявки в LINKS-режиме, ожидаемый `cloud_link != None` в БД.
   - smoke-тест на сценарий «при `LINKS` файл вложением отвергается с
     просьбой прислать ссылку».
   - проверка реестра: при `intake_mode = LINKS` поле №13 = URL
     (`view_command_or_link` уже это умеет, но нужен e2e-тест с реальным
     `Application.cloud_link`).

**Где в ТЗ.**

- §33.6 — режимы FILES/LINKS, переключение, контракт хранения.
- §25.1 поле №13 «Команда/ссылка просмотра файлов».
- §33.6.3 — поведение реестра в смешанном массиве FILES + LINKS.
- §28.1 — авто-переключение в LINKS при заполнении диска ≥
  `DISK_BLOCK_PCT`.

**Не блокирует Wave 4 deploy.** До выкатки LINKS-UX модератор/админ
оставляет `INTAKE_MODE_DEFAULT=files`, при заполнении диска авто-переход
в LINKS приведёт к мгновенной деградации UX (родителю всё равно покажут
«загрузите файл»), поэтому реальный сценарий до релиза LINKS-UX —
держать диск ниже `DISK_BLOCK_PCT` или ставить `intake_mode=links`
вручную и сообщать заявителям о ссылочном режиме другим каналом.

**Оценка.** ~0.5–1 день: 2 хендлера, 1 state, обновление текстов,
2–3 теста. Развязок с другими ветками нет.
