# Документация kids_ai

> **Эта папка — единственный источник правды по проекту.**
> `docs/history/` — исторический snapshot стартовых артефактов (включая
> первое техническое задание `ТЗ.md` и сопутствующие материалы). К нему
> **не возвращаемся и не правим**: любые изменения требований или
> поведения бота фиксируются прямо здесь, в актуальных файлах
> `docs/*.md`.

## Карта документации

| Документ | Назначение |
|----------|------------|
| [`architecture.md`](architecture.md) | Стек, структура модулей, lifespan, FSM, навигация, модель данных, контракты сервисов |
| [`deployment.md`](deployment.md) | Переменные окружения, docker-compose, offline-сборка, мониторинг, troubleshooting, бэкапы |
| [`registry-spec.md`](registry-spec.md) | Контракт XLSX-выгрузок `/export` и `/export_shortlist` (состав листов, колонки, форматирование, имя файла) |
| [`testing.md`](testing.md) | Как запускать `pytest`, что покрыто автотестами, ручной приёмочный чек-лист |
| [`backlog.md`](backlog.md) | Отложенные задачи (что сознательно не вошло в текущий релиз) |
| [`history/`](history/) | Архив: стартовое `ТЗ.md`, исходные «Вопросы…/Ответы разработчика». Read-only, не источник правды |

## Как менять документацию

1. Любые изменения требований, контракта команд бота, поведения сервисов
   или формата артефактов (Excel-реестр, имена файлов, тексты сообщений)
   фиксируются **прямо в соответствующем актуальном файле** `docs/*.md`.
2. `docs/history/ТЗ.md` **не правим** — это исторический snapshot. Если
   старая формулировка из ТЗ помогает объяснить дизайн-решение, можно
   однократно сослаться на неё фразой «исторический контекст — см.
   [`docs/history/ТЗ.md`](history/ТЗ.md)». Без такой пометки ссылок на
   ТЗ в актуальной документации быть не должно.
3. Для конкретных правил кода — см. `.cursor/rules/*.mdc` (логирование,
   commit-сообщения, навигация, **тексты сообщений** `user-messages.mdc`).
   Документация и правила не дублируются.
4. После значимых изменений в коде проверяй, не нужно ли обновить
   `architecture.md` (модули, модель данных) или `deployment.md`
   (env-vars, docker, мониторинг). Чек-лист — в скилле
   `.cursor/skills/project-docs/SKILL.md`.

## Принципы

- Документация формулирует требования и поведение **от первого лица**
  («Возрастных категорий три: 0–6, 7–12, 13–18»), а не отсылает к ТЗ
  («см. ТЗ §9»). Параграфных ссылок `§N.M` в актуальных доках быть не
  должно — внутренние перекрёстные ссылки делаются на якоря текущих
  документов (`architecture.md#fsm`, `deployment.md#scheduler` и т. п.).
- Если требование нужно описать подробнее — описываем его прямо здесь,
  а не отправляем читателя в `docs/history/`.
- Микро-коммиты, Conventional Commits (см. `.cursor/rules/conventional-commits.mdc`).
