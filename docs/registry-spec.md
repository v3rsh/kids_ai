## 1. Общие принципы

1. **Источник правды — БД** (`Application`, `ApplicationFile`, `JuryVote`, `JuryMember`, `JuryRound`).
2. **Никакой записи на диск.** Функции `build_registry_xlsx() -> bytes` и `build_shortlist_xlsx() -> bytes` собирают XLSX в `io.BytesIO` и возвращают `bytes`. Хендлер B3 отдаёт их в чат attachment'ом и забывает (§25.4).
3. **Никакого пересоздания на события.** Файл генерируется только в момент вызова `/export` / `/export_shortlist`.
4. **Производительность:** один SQL-запрос с `selectinload` за всё (заявки + файлы); отдельный single-pass-запрос за голосами с JOIN на `JuryMember` и `JuryRound`. N+1 запрещён (см. `.cursor/rules/performance.mdc`).
5. **Логирование:** `from loguru import logger`; на вход — `INFO`, на выход — `INFO` с длительностью и числом строк.

### 1.6. Пулы и возрастные категории — никакого хардкода в реестре

Состав возрастных категорий и число пулов на момент написания спецификации **уточняются** (изначально предполагалось 4 категории и 12 пулов; обсуждается переход на 3 категории и 9 пулов — это зона Wave 0 / ветки C, не ветки E). Чтобы реестр пережил любое изменение, в `app/services/registry.py` и в этой спецификации действуют следующие правила:

1. **Перебор пулов** в `build_shortlist_xlsx()` идёт **только** через `services.pools.all_pools()`. Жёсткое число (12, 9 и т. д.) ни в коде, ни в текстах спецификации не зашивается. Если в `services.pools.all_pools()` пулов 0 — лист `Шорт-лист` собирается с одной строкой заголовков и подписью «Нет завершённых пулов».
2. **Значения возрастных категорий** для колонок «Возрастная категория» (поле №8 основного листа и поле №6 шорт-листа) **берутся из `Application.age_category.value`** (т. е. из значения enum `AgeCategory` модели). Никаких литералов `"4–6"` / `"7–10"` / `"11–13"` / `"14–18"` в реестровом коде. Когда ветка C/Wave 0 поправит enum — реестр продолжит работать.
3. **Заголовки разделителей** в шорт-листе формируются как `f"{pool.track.value} / {pool.age_category.value}"` — тоже через `.value` enum'ов, не через хардкод-словарь.
4. **Имена колонок жюри** (§2.3.1) зависят от `JuryMember.full_name` и числа фактически проведённых раундов в данных (`SELECT DISTINCT round_no FROM jury_rounds WHERE status IN (CLOSED, DRAWN_BY_LOT)`); число судей и раундов не предполагается фиксированным.

Это правило фиксируется в спеке и проверяется при ревью: если в `app/services/registry.py` появляется литерал `12`, `9`, `"4–6"`, `["Традиционное рисование", "ИИ-рисунок", ...]` и т. п. — это регресс.

#### 2.2.2. Поле №13 «Команда/ссылка просмотра файлов» (Q9, решение: вариант A — одна колонка)

Содержимое колонки №13 — одно строковое значение, формируется по правилу:

```python
def field_13_view_command_or_link(app: Application) -> str:
    if app.intake_mode is IntakeMode.LINKS:
        return app.cloud_link or ""  # URL папки участника
    return f"/files {app.br_id}"      # команда модератора в чате
```

- Никаких 13a/13b: ТЗ §25.1 нумерует это поле как одно («№13 Команда/ссылка просмотра файлов»), §33.6.3 явно говорит «в реестре заменяется на столбец «Ссылка на папку» (для записей, поданных в режиме ссылок)» — это **тот же** столбец, разное содержимое.
- В реестре будет **смешанный массив** (часть заявок в режиме `files`, часть — в `links` после переключения по §33.6 / §28.1) — это нормально и предусмотрено ТЗ.
- Если `intake_mode is LINKS` и при этом `cloud_link` пустой (что не должно происходить в нормальном flow ветки A, но защищаемся) — пишем пустую строку, чтобы не падать.
- В шорт-листе (§3.1 поле №10) — то же правило, та же helper-функция.

#### 2.2.1. Формат даты подачи (Q3, решение: вариант A — ISO-строка)

Поле №2 «Дата и время подачи».

- **Заголовок колонки:** `Дата и время подачи (Europe/Moscow)`.
- **Тип ячейки в openpyxl:** строка (`str`), **не** native `datetime`. Это намеренно — `Application.created_at` хранится в UTC; перед записью переводится в `Europe/Moscow` через `ZoneInfo("Europe/Moscow")` и форматируется как `f"{dt:%Y-%m-%d %H:%M}"`. Хранение строкой исключает риск таймзонной путаницы при открытии в Excel/LibreOffice, который не понимает tz-aware `datetime`.
- **Формат значения:** `2026-06-01 14:32` (ISO без секунд и без зоны в самом значении). Тажа форма используется в `meta.txt` (§23.2) — единообразие.
- **Сортировка лексикографическая = сортировка по дате** для шаблона `YYYY-MM-DD HH:MM` — корректна без дополнительных усилий.

### 2.3. Лист `Голосование жюри` — порядок колонок

Фиксированные колонки слева:
1. `ID заявки` (`Application.br_id`).
2. `Трек` (`Application.track.value`).
3. `Возрастная категория` (`Application.age_category.value`).

Далее — динамические колонки на каждого активного `JuryMember` × фактически проведённый раунд (1..max_round в данных по конкретному пулу). Если в пуле прошёл только раунд 1 — колонки для раундов 2 и 3 у работ этого пула остаются пустыми.

#### 2.3.0. Какие заявки выводятся на этот лист (Q10, решение: вариант A — только с `JuryVote`)

Лист `Голосование жюри` содержит **только заявки, у которых есть хотя бы одна запись в `jury_votes`** (любого состояния — `DRAFT` или `SUBMITTED`). Заявки, не дошедшие до жюри (статус модерации `на_модерации` / `нужно_исправить` / `отклонено`, либо `допущено` но без открытого пула / без раундов), на этом листе **не выводятся**.

**SQL-эскиз (single-pass):**

```python
stmt = (
    select(Application)
    .where(
        Application.id.in_(
            select(JuryVote.application_id).distinct()
        )
    )
    .order_by(Application.br_id.asc())
)
```

Аргументы:

- лист — это **детализация по голосам**, а не «полный список заявок». Если у заявки нет ни одного голоса — её там быть не должно по смыслу;
- общий вид «все заявки + их статус жюри» уже есть в основном листе `Реестр` (поле №16 «Статус жюри» + поля 23–29 агрегатов). Дубли в `Голосование жюри` избыточны;
- при ожидаемых ~100 заявках до жюри дойдёт большинство, но 20–30 % будут отклонены / на исправлении — выводить их строками со 100 % пустыми динамическими колонками значит просадить читаемость на треть;
- решение «по наличию `JuryVote`, а не по статусу жюри» намеренно: даёт корректный лист даже в полузакрытых состояниях (раунд открыт, но никто ещё не отправил оценки — лист пуст; первый судья поставил `DRAFT` — заявка появилась со всеми пустыми ячейками голосов).

#### 2.3.1. Формат имён динамических колонок (Q1, решение: вариант C)

Шаблон: `<Фамилия в транслитерации>.<инициал имени в транслитерации>_r<номер_раунда>`.

**Примеры:**

| `full_name` в `JuryMember` | Заголовок колонки (раунд 1) |
|----------------------------|------------------------------|
| `Винокурова Екатерина` | `Vinokurova.E_r1` |
| `Иванова Анастасия` | `Ivanova.A_r1` |
| `Кулаева Варвара` | `Kulaeva.V_r1` |
| `Юдин Юрий` | `Iudin.I_r1` |
| `Щербак Эльвира` | `Shcherbak.E_r1` |

> ⚠️ **Уточнение шаблона относительно исходного варианта C `E.Vinokurova_r1`.** В образце инициал стоит перед фамилией (`E.Vinokurova`), но для удобной сортировки модератором по фамилии в спецификации зафиксирован порядок `Фамилия.И_rN`. Если такое отклонение неприемлемо — нужно вернуться к Q1 и переутвердить.

**Разбор `full_name`:**

1. `tokens = full_name.strip().split()` (по любому whitespace).
2. Если `len(tokens) >= 2`: `surname = tokens[0]`, `first_initial = tokens[1][0]`.
3. Если `len(tokens) == 1` или после транслитерации получается пустая строка — **fallback**: имя колонки = `<full_name_без_изменений>_r<N>` (без транслитерации, кириллицей).

**Таблица транслитерации — паспортный стандарт ICAO Doc 9303 (она же действующая редакция Приказа МВД РФ № 889 / Приказа МИД РФ № 4271 от 2014):**

| Буква | Транслит | Буква | Транслит | Буква | Транслит |
|-------|----------|-------|----------|-------|----------|
| А | A | Л | L | Х | KH |
| Б | B | М | M | Ц | TS |
| В | V | Н | N | Ч | CH |
| Г | G | О | O | Ш | SH |
| Д | D | П | P | Щ | SHCH |
| Е | E | Р | R | Ъ | IE |
| Ё | E | С | S | Ы | Y |
| Ж | ZH | Т | T | Ь | (опускается) |
| З | Z | У | U | Э | E |
| И | I | Ф | F | Ю | IU |
| Й | I | | | Я | IA |
| К | K | | | | |

> Регистр результата: первая буква каждого токена в **Title-case**, остальные — в lower-case (`KH` → `Kh`, `SHCH` → `Shch`). Это даёт чтение, привычное модератору: `Shcherbak` вместо `SHCHERBAK`.

**Алгоритм (упрощённо):**

```python
def transliterate_passport(text: str) -> str:
    table = {"А": "A", "Б": "B", ..., "Я": "IA"}  # см. таблицу выше
    out = []
    for ch in text:
        if ch.isupper():
            out.append(table.get(ch, ch))
        elif ch.islower():
            up = ch.upper()
            mapped = table.get(up, ch)
            out.append(mapped.lower())
        else:
            out.append(ch)
    return "".join(out)


def jury_column_name(full_name: str, round_no: int) -> str:
    tokens = full_name.strip().split()
    if len(tokens) >= 2 and tokens[0] and tokens[1]:
        surname = transliterate_passport(tokens[0])
        initial = transliterate_passport(tokens[1][0])
        if surname and initial:
            return f"{surname.title()}.{initial.upper()}_r{round_no}"
    # fallback на исходное ФИО без транслитерации
    return f"{full_name.strip()}_r{round_no}"
```

#### 2.3.2. Формат значений голоса (Q2, решение: вариант C — числовой)

| Состояние в БД | Значение в ячейке |
|----------------|-------------------|
| `JuryVote.state = SUBMITTED`, `JuryVote.vote = YES` (Достоин) | `1` |
| `JuryVote.state = SUBMITTED`, `JuryVote.vote = NO` (Не достоин) | `0` |
| Работа не попала в данный раунд по пулу (раунд не открывался / открывался без этой заявки) | пусто |
| Судья назначен на пул, но не отправил голос до закрытия раунда (нет `SUBMITTED`-записи; `DRAFT` тоже считается «не отправил») | пусто |

> ⚠️ **Дифференциация двух «пустых» причин.** Пустая ячейка одинаково означает как «работа не попала в этот раунд», так и «судья не отправил голос». Чтобы их различить, модератор смотрит на колонку **«Итоговый раунд» (поле №26)** в основном листе `Реестр`:
>
> - если `№26 < N` — работа в раунд N просто не попала;
> - если `№26 >= N` — работа в раунд N попала, и пустота означает «судья не отправил голос».
>
> Это решение принято для компактности листа (числовой формат `1`/`0`/пусто экономит ширину колонок при 5–15 судьях × до 3 раундов). При необходимости можно вернуться к Q2 и переключиться на текстовый формат (`Достоин`/`Не достоин`/`—`/`не голосовал`).

**Тип ячейки в openpyxl:** число (`int`), а не строка — модератор сможет фильтровать/суммировать средствами Excel без преобразования. Заголовок колонки остаётся текстовым (см. §2.3.1).

> Цель документа — «вот топ-N по пулу, идём в производство». Лист должен помещаться по ширине на одну страницу A4 в landscape для печати — 13 колонок укладываются.

### 3.2. Разбивка по пулам (Q6.2, решение: вариант X — один лист с разделителями)

Файл шорт-листа содержит **ровно один лист** с именем `Шорт-лист`. Внутри листа строки группируются по пулам в порядке, который отдаёт `services.pools.all_pools()`. Пул = пара `(track, age_category)`; число и состав пулов **не зашиваются в `registry.py`** — это контракт ветки C (§1.6).

**Структура листа:**

```
строка 1                   — заголовки колонок (13 колонок из §3.1), серая заливка темнее, жирный шрифт
строка 2                   — разделитель пула: «Трек / Возрастная категория»
                              (merged через все 13 колонок, светло-серая заливка, жирный)
строки 3..M+2              — топ-N работ первого пула, отсортированы по pool_position ASC
строка M+3                 — разделитель следующего пула
строки M+4..               — топ-N работ второго пула
...
последняя строка           — топ-N последнего пула (по порядку services.pools.all_pools())
```

Внутри каждого пула сортировка — `ORDER BY pool_position ASC, br_id ASC` (стабильно при равных позициях, см. §35.5 — жребий проставляется на финальном раунде).

**Пулы без работ в топ-10** (например, в пуле было меньше N заявок или жюри не дошло до этого пула) выводятся со строкой-разделителем и **одной строкой-подписью** «Нет работ в топ-N для этого пула» (merged через 13 колонок, italic, светло-серая заливка). Это лучше, чем молча пропускать пул — модератор сразу видит, что пул учтён, но пуст.

**Один лист, а не N** — потому что:

- общий объём — обычно `(число пулов) × top_N` строк (для `top_N=10` и 9 пулов это 90 строк, для 12 — 120; влезает на 2–3 страницы A4), вкладки избыточны;
- удобно копировать «целиком» в письмо/презентацию по итогам жюри;
- 9–12 вкладок резко ухудшают навигацию (модератор листает табы), один лист с явными разделителями читается линейно;
- если позже понадобятся отдельные листы для дизайнера/мерч-команды — это точечная правка `app/services/registry.py` без поломки контракта (имя файла, имя листа `Шорт-лист`, число колонок сохраняются).

## 4. Имя отправляемого файла (Q4, решение: вариант A — детальное)

### 4.1. Шаблоны

| Команда | Шаблон имени | Пример |
|---------|--------------|--------|
| `/export` | `registry_BR-{YEAR}_{YYYY-MM-DD}_{HH-MM}.xlsx` | `registry_BR-2026_2026-06-15_14-32.xlsx` |
| `/export_shortlist` | `shortlist_BR-{YEAR}_{YYYY-MM-DD}_{HH-MM}.xlsx` | `shortlist_BR-2026_2026-06-15_14-32.xlsx` |

### 4.2. Источники подстановки

- `YEAR` — `config.COMPETITION_YEAR` (по умолчанию `2026`, переопределяется через env);
- `YYYY-MM-DD` и `HH-MM` — момент вызова команды модератором, **в Europe/Moscow**: `datetime.now(ZoneInfo("Europe/Moscow"))`;
- разделитель в `HH-MM` — дефис `-` (не двоеточие), потому что `:` в имени файла недопустим в Windows.

### 4.3. Публичный helper в `app/services/registry.py`

Чтобы B3 и любой другой потенциальный потребитель имели единый источник правды по формату имени, сервис экспортирует:

```python
def registry_export_filename(
    kind: Literal["registry", "shortlist"],
    now_msk: datetime | None = None,
) -> str:
    """Имя файла для on-demand выгрузки (§4 docs/registry-spec.md).

    Аргументы:
        kind: тип выгрузки — основной реестр или шорт-лист.
        now_msk: момент вызова в Europe/Moscow; если None — берётся
            datetime.now(ZoneInfo("Europe/Moscow")).
    """
```

### 4.4. ⚠️ Известное расхождение с уже написанным B3

В `app/handlers/moderator_export.py:_xlsx_filename()` (Wave 2 / ветка B3) сейчас сформирован собственный шаблон `{prefix}_BR-{COMPETITION_YEAR}_{today}.xlsx`, **без** `HH-MM` и **без** явной зоны (`datetime.now()` без `ZoneInfo`). Это не совпадает с зафиксированным в §4.1.

Решение: при подключении ветки E (когда `build_registry_xlsx()` и helper будут реализованы) B3-хендлер должен перейти на `registry.registry_export_filename(...)` из ветки E. Правка `app/handlers/moderator_export.py` находится **вне зоны ветки E** — несоответствие фиксируется здесь, чтобы:

1. барьер-агент Wave 3 поймал расхождение на интеграции;
2. либо ветка B сделала точечный коммит с импортом helper'а после публикации сервиса.

Альтернатива (если ваша воля — оставить как у B3, без `HH-MM`): сообщите, я переоткрою Q4 и пересажу спецификацию обратно на короткий шаблон.

## 5. Сортировка и фильтры

### 5.1. Сортировка строк в листе `Реестр` (Q5, решение: вариант D — по `br_id` ASC)

`ORDER BY application.br_id ASC` — единственный гарантированно стабильный и детерминированный порядок (br_id уникален и монотонно возрастает по §20).

- Реестр — юридический документ конкурса (хранится до 30 сентября, §29.1); у документа должен быть фиксированный порядок строк, не зависящий от UTC/таймзоны и от микросекундной разрешающей способности `created_at`.
- Строки листа `Голосование жюри` (§2.3) сортируются по тому же ключу — модератор может визуально сопоставить две вкладки.
- Если модератору нужно «новые сверху» — он включает автофильтр (§5.3) и переворачивает сортировку в один клик.

**SQL-эскиз:**

```python
stmt = (
    select(Application)
    .options(selectinload(Application.files))
    .order_by(Application.br_id.asc())
)
```

### 5.2. Защита листов от редактирования (Q7, решение: вариант A — НЕ включается)

Sheet protection (`worksheet.protection.enable()`) **не применяется** ни к одному листу (`Реестр`, `Голосование жюри`, `Шорт-лист`).

Аргументы:

- §25.4 ТЗ: «защищать ничего не требуется»; «изменения в скачанной модератором копии не влияют ни на что»;
- источник правды — БД; любые правки в файле теряются при следующем `/export`;
- защита без пароля даёт ложное чувство «защищённости», но снимается в один клик;
- защита с паролем требует управления секретом, что выходит за рамки MVP и противоречит хранению пароля в коде/логах;
- openpyxl с включённой защитой иногда даёт несовместимость со старыми версиями LibreOffice.

## 6. Локализация заголовков (Q11, решение: вариант A — только русские)

Все заголовки колонок основного листа `Реестр` и шорт-листа `Шорт-лист` — **только русские**, дословно по §25.1 ТЗ. Технические имена в скобках (`(br_id)`, `(parent_full_name)`, ...) **не добавляются**.

Аргументы:

- §25.1 ТЗ — источник правды по составу полей; модератор Excel'а техническими именами не пользуется;
- русские заголовки короче (`ID заявки` = 9 vs `ID заявки (br_id)` = 17), укладываются в ширины §5.3.1;
- источник правды — БД (§25.4); обратный импорт из Excel не предусмотрен → технические имена в шапке не нужны.

**Исключение — лист `Голосование жюри`:** динамические колонки судей остаются в **латинице** (`Vinokurova.E_r1`) по §2.3.1 — это намеренный компромисс (короткий стабильный «код колонки»).

## 7. Производительность и кэш (Q12, решение: вариант A — никакого кэша)

### 7.1. Никакого кэша

Каждый вызов `/export` и `/export_shortlist` собирает XLSX **с нуля** из актуального состояния БД. In-memory кэш, Redis-кэш, фоновая сборка по cron — **не применяются**:

- §25.4 ТЗ: «генерация только по запросу» (любой кэш — это автопересборка по таймеру / по первому запросу, что противоречит модели);
- модератор всегда получает актуальный снимок (нет инвалидации, нет вопроса «почему статус другой, чем в /find»);
- архитектурно проще (нет состояния в памяти, нет фонового job'а, нет ключей в Redis);
- pybotx-приложение однопоточное в нашем deployment'е — конкуренции на сборку нет.

### 7.2. Логирование длительности

Сборка обоих файлов оборачивается в замер времени; на выход в `INFO` пишется длительность и число строк:

```python
import time
from loguru import logger

t0 = time.perf_counter()
logger.info("registry build start", kind="registry")
# ... сборка ...
duration_ms = (time.perf_counter() - t0) * 1000
logger.info(
    "registry build done",
    kind="registry",
    rows=N_main,
    cols=N_main_cols,
    detail_rows=N_jury_detail,
    duration_ms=round(duration_ms, 1),
)
if duration_ms > 5000:
    logger.warning(
        "registry build exceeded soft threshold",
        duration_ms=round(duration_ms, 1),
        threshold_ms=5000,
        hint="см. Q12 в docs/registry-spec.md — рассмотреть кэш на короткий TTL",
    )
```

### 7.3. Алерт-порог 5 секунд

**Soft-порог 5000 мс.** При первом превышении на проде — Wave 3 (или ветка E повторно) возвращается к Q12 и реализует кэш на короткий TTL (вариант B из исходного списка).

Smoke-проверка на 1000 синтетических заявок — добавляется в Wave 3 (тесты, §30.1). Ожидаемое время:

- основной лист: ~500–2000 мс (1000 строк × 29 колонок + 1000 строк детализации × (3 фикс + ~10 динамических));
- шорт-лист: 90–120 строк, ~50–150 мс.

Если smoke покажет p95 > 5000 мс — реализуем кэш TTL=60s (in-memory `dict[kind, tuple[float, bytes]]`, легко вставляется поверх текущих `build_*_xlsx`).
