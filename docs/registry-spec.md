## 1. Общие принципы

1. **Источник правды — БД** (`Application`, `ApplicationFile`, `JuryVote`, `JuryMember`, `JuryRound`).
2. **Никакой записи на диск.** Функции `build_registry_xlsx() -> bytes` и `build_shortlist_xlsx() -> bytes` собирают XLSX в `io.BytesIO` и возвращают `bytes`. Хендлер B3 отдаёт их в чат attachment'ом и забывает (§25.4).
3. **Никакого пересоздания на события.** Файл генерируется только в момент вызова `/export` / `/export_shortlist`.
4. **Производительность:** один SQL-запрос с `selectinload` за всё (заявки + файлы); отдельный single-pass-запрос за голосами с JOIN на `JuryMember` и `JuryRound`. N+1 запрещён (см. `.cursor/rules/performance.mdc`).
5. **Логирование:** `from loguru import logger`; на вход — `INFO`, на выход — `INFO` с длительностью и числом строк.

## 2. Команда `/export` — основной реестр

### 2.1. Состав листов

1. `Реестр` — поля 1–22 (§25.1) + агрегаты 23–29 (§25.3.1). Одна строка = одна заявка.
2. `Голосование жюри` — детализация по членам жюри × раундам (§25.3.2). Одна строка = одна заявка, динамические колонки на каждого активного `JuryMember` × фактически проведённый раунд (1..max_round в данных).

### 2.3. Лист `Голосование жюри` — порядок колонок

Фиксированные колонки слева:
1. `ID заявки` (`Application.br_id`).
2. `Трек` (`Application.track.value`).
3. `Возрастная категория` (`Application.age_category.value`).

Далее — динамические колонки на каждого активного `JuryMember` × фактически проведённый раунд (1..max_round в данных по конкретному пулу). Если в пуле прошёл только раунд 1 — колонки для раундов 2 и 3 у работ этого пула остаются пустыми.

#### 2.3.1. Формат имён динамических колонок (Q1, решение: вариант C)

Шаблон: `<Фамилия в транслитерации>.<инициал имени в транслитерации>_r<номер_раунда>`.

**Примеры:**

| `full_name` в `JuryMember` | Заголовок колонки (раунд 1) |
|----------------------------|------------------------------|
| `Винокурова Екатерина` | `Vinokurova.E_r1` |
| `Иванова Анастасия` | `Ivanova.A_r1` |
| `Кулаева Варвара` | `Kulaeva.V_r1` |
| `Юдин Юрий` | `Iudin.I_r1` |
| `Щербак Эльвира` | `Shcherbak.E_r1` |

> ⚠️ **Уточнение шаблона относительно исходного варианта C `E.Vinokurova_r1`.** В образце инициал стоит перед фамилией (`E.Vinokurova`), но для удобной сортировки модератором по фамилии в спецификации зафиксирован порядок `Фамилия.И_rN`. Если такое отклонение неприемлемо — нужно вернуться к Q1 и переутвердить.

**Разбор `full_name`:**

1. `tokens = full_name.strip().split()` (по любому whitespace).
2. Если `len(tokens) >= 2`: `surname = tokens[0]`, `first_initial = tokens[1][0]`.
3. Если `len(tokens) == 1` или после транслитерации получается пустая строка — **fallback**: имя колонки = `<full_name_без_изменений>_r<N>` (без транслитерации, кириллицей).

**Таблица транслитерации — паспортный стандарт ICAO Doc 9303 (она же действующая редакция Приказа МВД РФ № 889 / Приказа МИД РФ № 4271 от 2014):**

| Буква | Транслит | Буква | Транслит | Буква | Транслит |
|-------|----------|-------|----------|-------|----------|
| А | A | Л | L | Х | KH |
| Б | B | М | M | Ц | TS |
| В | V | Н | N | Ч | CH |
| Г | G | О | O | Ш | SH |
| Д | D | П | P | Щ | SHCH |
| Е | E | Р | R | Ъ | IE |
| Ё | E | С | S | Ы | Y |
| Ж | ZH | Т | T | Ь | (опускается) |
| З | Z | У | U | Э | E |
| И | I | Ф | F | Ю | IU |
| Й | I | | | Я | IA |
| К | K | | | | |

> Регистр результата: первая буква каждого токена в **Title-case**, остальные — в lower-case (`KH` → `Kh`, `SHCH` → `Shch`). Это даёт чтение, привычное модератору: `Shcherbak` вместо `SHCHERBAK`.

**Алгоритм (упрощённо):**

```python
def transliterate_passport(text: str) -> str:
    table = {"А": "A", "Б": "B", ..., "Я": "IA"}  # см. таблицу выше
    out = []
    for ch in text:
        if ch.isupper():
            out.append(table.get(ch, ch))
        elif ch.islower():
            up = ch.upper()
            mapped = table.get(up, ch)
            out.append(mapped.lower())
        else:
            out.append(ch)
    return "".join(out)


def jury_column_name(full_name: str, round_no: int) -> str:
    tokens = full_name.strip().split()
    if len(tokens) >= 2 and tokens[0] and tokens[1]:
        surname = transliterate_passport(tokens[0])
        initial = transliterate_passport(tokens[1][0])
        if surname and initial:
            return f"{surname.title()}.{initial.upper()}_r{round_no}"
    # fallback на исходное ФИО без транслитерации
    return f"{full_name.strip()}_r{round_no}"
```

#### 2.3.2. Формат значений голоса (Q2, решение: вариант C — числовой)

| Состояние в БД | Значение в ячейке |
|----------------|-------------------|
| `JuryVote.state = SUBMITTED`, `JuryVote.vote = YES` (Достоин) | `1` |
| `JuryVote.state = SUBMITTED`, `JuryVote.vote = NO` (Не достоин) | `0` |
| Работа не попала в данный раунд по пулу (раунд не открывался / открывался без этой заявки) | пусто |
| Судья назначен на пул, но не отправил голос до закрытия раунда (нет `SUBMITTED`-записи; `DRAFT` тоже считается «не отправил») | пусто |

> ⚠️ **Дифференциация двух «пустых» причин.** Пустая ячейка одинаково означает как «работа не попала в этот раунд», так и «судья не отправил голос». Чтобы их различить, модератор смотрит на колонку **«Итоговый раунд» (поле №26)** в основном листе `Реестр`:
>
> - если `№26 < N` — работа в раунд N просто не попала;
> - если `№26 >= N` — работа в раунд N попала, и пустота означает «судья не отправил голос».
>
> Это решение принято для компактности листа (числовой формат `1`/`0`/пусто экономит ширину колонок при 5–15 судьях × до 3 раундов). При необходимости можно вернуться к Q2 и переключиться на текстовый формат (`Достоин`/`Не достоин`/`—`/`не голосовал`).

**Тип ячейки в openpyxl:** число (`int`), а не строка — модератор сможет фильтровать/суммировать средствами Excel без преобразования. Заголовок колонки остаётся текстовым (см. §2.3.1).
