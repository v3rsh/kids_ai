# Вопросы для технарей компании

Что нужно прояснить с административной командой контура (админы NextCloud, eXpress, DevOps), чтобы корректно собрать и развернуть бота во внутреннем контуре.

> **Контекст.** Проект «Безопасные рисунки» подтверждён, разработка начинается. Бот будет поставлен в виде заранее собранного на Mac Docker-образа (архив с образами и зависимостями привезу руками). Все библиотеки и базовые образы упаковываются в архив на этапе сборки — у бота **нет доступа в интернет** на сервере. Postgres и Redis поднимаются рядом с ботом в том же `docker-compose`. Часовой пояс — **Europe/Moscow** на всех уровнях.

> **TL;DR — что критично узнать в первую очередь:**
>
> 1. **NextCloud — есть и доступен?** Это блокер всей разработки. Если NextCloud недоступен — нужно как можно раньше инициировать согласование Beeline Cloud / S3 с Департаментом защиты данных (раздел 8 ответов компании, раздел 33 ТЗ). Все вопросы блока «Хранилище» — приоритет №1.
> 2. **Параметры подключения к CTS eXpress** — без них бот не запустится.
> 3. **ОС и архитектура сервера** — нужно для кросс-сборки образа на Mac.
> 4. **Машины модераторов и HEIC** — влияет на UX просмотра работ.

---

## 🔴 Блок 1. Хранилище (NextCloud) — приоритет №1

Компания подтвердила, что основной сценарий — NextCloud, но просила «как можно раньше узнать, если возникнут проблемы». Нужны точные параметры до старта разработки модуля хранилища.

### 1.1. Доступность и адрес

- NextCloud в контуре **есть** или его нужно развернуть?
- Внутренний URL NextCloud-сервера.
- Версия NextCloud (от этого зависит набор поддерживаемых WebDAV-операций).

### 1.2. Технический пользователь

- Создать технического пользователя под бота (логин, например `safe_drawings_bot`).
- App-password для этого пользователя (или возможность сгенерировать самостоятельно через профиль).
- Корневая папка, в которую бот пишет: `Безопасные рисунки/` — нужно создать заранее или бот создаёт сам?
- Права тех. пользователя: чтение / запись / создание директорий / **MOVE между директориями** (критично для переноса в `99_Отклонено/`).

### 1.3. Поддержка операций

- Подтвердите, что доступны WebDAV-методы: `MKCOL`, `PUT`, `GET`, `MOVE`, `DELETE`, `PROPFIND`.
- Поддерживаются ли **кириллические** имена файлов и папок без проблем (мы делаем имена вида `BR-2026-0001_Иванова_Анна_Маша/`)? Были ли в контуре известные проблемы с UTF-8 в путях?
- Лимит размера файла на стороне NextCloud (мы уперлись в 20 МБ на стороне бота, но если у NextCloud меньше — нужно знать).

### 1.4. Публичные ссылки на папки

- Может ли тех. пользователь шарить папки публичной ссылкой через OCS Share API (для удобства модераторов)?
- Есть ли ограничения по сроку жизни / паролю на ссылку?
- Альтернатива: модераторы открывают папки через свой авторизованный доступ к NextCloud — тогда публичные ссылки не нужны.

### 1.5. Сертификаты

- NextCloud работает по HTTPS с сертификатом внутреннего корпоративного CA или с самоподписным?
- Если да — пришлите корневой сертификат (PEM), чтобы клиент бота доверял серверу.

### 1.6. Резервный сценарий (если NextCloud отпадает)

Только на случай, если по итогам выяснится, что NextCloud недоступен:

- Что используется как сетевой диск в контуре альтернативно (Beeline Cloud Workspace, Synology, MinIO, обычная SMB-шара)?
- В случае Beeline Cloud Workspace — компания упомянула необходимость согласования с Департаментом защиты данных. Кто инициирует согласование?

## 🟡 Блок 2. eXpress / BotX

### 2.1. Подключение бота к CTS

- Адрес on-premise CTS-сервера, к которому будем коннектиться.
- Адрес Bot Storage (если используется отдельный).
- Регистрируем нового бота под наш проект или есть уже выделенный? Если регистрируем — пришлите `bot_id`, `secret_key`, `host`.
- На какой URL CTS будет слать вебхуки боту (адрес и порт нашего контейнера во внутренней сети)?

### 2.2. Лимиты вложений на CTS

- Какой максимальный размер одного вложения настроен на сервере (по умолчанию обычно 25 МБ)? Должен быть **не меньше 20 МБ** (наш лимит на стороне бота).
- Есть ли лимит на количество вложений в одном сообщении / на количество сообщений в минуту?

### 2.3. Поддержка форматов вложений

- Подтвердите, что CTS пропускает все нужные нам форматы файлов: **JPG, JPEG, PNG, HEIC, WEBP, PDF**.
- Есть ли в CTS «чёрный список» расширений или MIME-типов?

### 2.4. Сертификаты CTS

- CTS работает по HTTPS с самоподписным сертификатом или с сертификатом внутреннего корпоративного CA?
- Если да — нужен корневой сертификат (PEM).

## 🟡 Блок 3. Сервер развёртывания

### 3.1. ОС и Docker

- Какая ОС на сервере (Ubuntu / RHEL / Astra Linux / Альт Линукс / другое)?
- Версия ОС.
- Установлен ли Docker и Docker Compose? Версии? (нужен Docker Compose v2.20+).
- Если Docker не установлен — кто его ставит, и можно ли поставить из локального репозитория (без интернета)?

### 3.2. Архитектура

- Архитектура процессора (`amd64` / `arm64`)?
- Нужно для правильной кросс-сборки на Mac (`docker build --platform linux/<arch>`).

### 3.3. Ресурсы

- Сколько ядер / памяти / диска можно выделить под `docker-compose`-стек (бот + Postgres + Redis)?
- Ориентировочная оценка с моей стороны: 2 vCPU, 4 ГБ RAM, 50 ГБ диска под данные на пиковую нагрузку конкурса.

### 3.4. Развёртывание

- Куда складывать архив с образами и `docker-compose.yml`? (например, `/opt/safe_drawings/`).
- Есть ли у меня прямой shell-доступ к серверу для запуска `docker compose up`, или передаю архив админам, и они разворачивают сами?

## 🟡 Блок 4. Сеть

### 4.1. Сетевая изоляция

- Подтверждаем: у контейнера бота **нет доступа в интернет**, только локальные адреса CTS и NextCloud.
- Нужен ли исходящий прокси для доступа к корпоративным внутренним адресам (если CTS / NextCloud в другой подсети)?

### 4.2. Входящие соединения

- Какие порты надо открыть со стороны CTS на наш контейнер (для приёма вебхуков)?
- Будем ли мы за реверс-прокси (nginx / Traefik), который терминирует TLS, или контейнер слушает напрямую?
- Если за реверс-прокси — кто его настраивает и какой адрес слушать на своём порту?

### 4.3. Резолвинг DNS

- Внутренние имена CTS и NextCloud резолвятся через корпоративный DNS, или нужно прописать их вручную в `/etc/hosts` контейнера?

## 🟡 Блок 5. Машины модераторов

В реестре будут лежать файлы, в т.ч. в формате **HEIC** (формат iPhone). Нужно понять, увидят ли их модераторы.

### 5.1. ОС модераторов

- На каких ОС работают модераторы (Екатерина Винокурова, Анастасия Иванова) — Windows 10 / Windows 11 / macOS / Linux?

### 5.2. Поддержка HEIC

- На Windows 10/11 для просмотра HEIC нужно установленное «HEIF Image Extension» от Microsoft. Установлено ли оно у модераторов корпоративно?
- Если **нет** — допустимо ли мне дополнительно создавать рядом превью в JPG (например, `BR-2026-0001_original.heic` + `BR-2026-0001_original_preview.jpg`)? Это +небольшой объём в хранилище, но снимет проблему просмотра.

### 5.3. Доступ модераторов к NextCloud

- Модераторы открывают папки через веб-интерфейс NextCloud, или у них примонтирован NextCloud как сетевой диск через клиент?
- Если сетевой диск — нет ли проблем с длиной пути на Windows (лимит 260 символов). Глубина нашего пути: ~6 уровней + длинное имя папки заявки.

## 🟢 Блок 6. Эксплуатация (необязательно для старта, но желательно)

### 6.1. Бэкапы

Файлы заявок и БД бэкапит сам разработчик (зона ответственности зафиксирована).

- Есть ли требование к месту хранения бэкапов (отдельный том / сетевой диск / S3-хранилище в контуре)?
- Можно ли использовать `pg_dump` по расписанию через `cron` внутри контейнера, или должна быть внешняя система резервного копирования?
- Срок хранения бэкапов?

### 6.2. Логи

- Куда писать логи бота — `stdout` контейнера + автоматический сбор (Loki / ELK / Graylog), или просто файлы на диске с ротацией?
- Срок хранения логов?
- Что в логах **нельзя** хранить (помимо очевидного — паролей)? Например, ФИО детей, контакты родителей. По нашему плану в логах будут только HUID и ID заявки, без чувствительных данных.

### 6.3. Мониторинг

- Нужен ли health-check эндпоинт (`/healthz`) для системы мониторинга? В каком формате (Prometheus-метрики / простой 200 OK)?
- Куда оповещать о падении бота — на тот же служебный чат в eXpress «Безопасные рисунки — модерация»?
