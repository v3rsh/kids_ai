# Ответы разработчика на вопросы из раздела 32 ТЗ

Подобные проекты я уже делал, ниже — мои ответы по каждому пункту раздела 32.

---

### 1. Можно ли технически принимать несколько файлов в одной заявке?

**Да.** eXpress нормально принимает несколько файловых сообщений подряд: бот переводит диалог в режим «жду файлы», накапливает их в черновике («файл 1 из 4 принят») и фиксирует заявку только после кнопки «Готово». Лимиты по треку из раздела 12 контролирует сам бот.

### 2. Можно ли принимать несколько сообщений от одного пользователя в рамках одной заявки?

**Да, это штатный сценарий.** Заявка собирается пошагово, ответы и файлы копятся в черновике. Если родитель закроет чат и вернётся позже — продолжит с того же шага.

### 3. Можно ли автоматически создавать папки в NextCloud / Билайн Облаке?

**Да.** В обоих случаях есть стандартный API. Закрываю это одним модулем хранилища внутри бота — когда заказчик подтвердит, какое именно облако в контуре, подключу нужный коннектор без переписывания логики.

### 4. Можно ли автоматически создавать структуру: день / трек / возраст / ID заявки?

**Да.** Структура из раздела 21.1 создаётся «по требованию» при поступлении первой заявки в нужную ветку. Пустые папки не плодим.

### 5. Можно ли автоматически формировать Excel-реестр со ссылками на папки?

**Да.** Источник правды — база бота. После каждого изменения бот пересобирает свежий `registry.xlsx` (все 22 колонки из раздела 25.1) и кладёт его в служебную подпапку конкурса в облаке. Ссылка на папку с файлами — кликабельна прямо из Excel.

> **Важно:** Excel — это **витрина для просмотра**, а не источник правды. Файл защищён от случайных правок (read-only protection + предупреждение в шапке листа). Любые изменения в данных делаются только через команды боту — иначе правки модератора в Excel при следующей перегенерации файла будут затёрты. Это сделано осознанно, чтобы исключить потерю данных и конфликты при одновременной работе.

### 6. Можно ли сделать модераторский доступ без полноценной админки?

**Да, и я предлагаю так и сделать — отдельную веб-админку не делаем.** Вся модерация идёт прямо в чате с ботом, никаких внешних интерфейсов:

- `/queue` — список заявок на модерации с инлайн-кнопками «Принять / На исправление / Отклонить»;
- `/find BR-2026-XXXX` — карточка заявки + кнопки смены статусов всех 4 групп (раздел 26);
- `/status …` — сменить статус (модерация / жюри / голосование / мерч);
- `/comment …` — добавить комментарий модератора;
- `/bulk_status …` — массовая смена статуса по фильтру (например, «принять все за вчера в треке X»);
- `/export` — прислать свежий XLSX в чат;
- `/stats` — статистика по разделу 28 ТЗ.

Список модераторов прошивается на старте. Модератор живёт в одном окне (eXpress), Excel открывает только для общего обзора. Это закрывает весь раздел 27 ТЗ без потерь.

### 7. Можно ли отправлять уведомления организаторам в служебный чат?

**Да.** Бот может писать и в групповой чат, и в личку. Удобнее всего — общий чат «Безопасные рисунки — модерация», куда добавляется бот и все получатели.

### 8. Есть ли ограничения по количеству файлов в одной заявке?

**Со стороны eXpress — практически нет** (только лимит на размер одного вложения, уточняется у админов CTS). Со стороны бота — действуют лимиты раздела 12 ТЗ: 1 файл для рисунка/коллажа/ИИ-рисунка/диптиха, 2–4 файла для поделки/3D/инсталляции. Лишнее бот не примет.

### 9. Можно ли принять повторную заявку и пометить её как новую версию?

**Да.** Каждая отправка — новая строка с собственным ID, ничего не перезаписывается. Бот сам ищет предыдущие заявки от того же родителя по тому же ребёнку и треку, проставляет «Возможный дубль» и «Связанная заявка» и подсвечивает это модератору. Флажок «Актуальная версия» проставляет модератор.

### 10. Можно ли автоматически создавать `description.txt` и `meta.txt`?

**Да.** Оба файла бот формирует и кладёт в папку заявки сразу после загрузки. Кодировка UTF-8 — на Windows у модератора всё открывается без «крякозябр».

### 11. Можно ли переименовывать файлы по заданному шаблону?

**Да**, по шаблонам раздела 22. Исходное имя файла сохраняется в `meta.txt` для возможности обратной связи с оригиналом.

### 12. В каком формате проще реализовать реестр: XLSX, CSV или внутренняя таблица?

**Связка из двух уровней:** источник правды — внутренняя БД бота (защищает от потери данных и конфликтов при одновременных заявках), витрина для модератора — автоматически обновляемый `registry.xlsx` в общей папке. CSV-экспорт добавляется одной командой при необходимости.

### 13. Что из описанного в ТЗ входит в MVP, а что вынесено в ручной процесс?

**В MVP бота входит всё, что описано в ТЗ** — проект небольшой, разрывать его на этапы смысла нет:

- меню и информационные тексты (разделы 6, 7);
- сценарий подачи заявки, выбор трека и категории (разделы 8–14);
- автопроверки и согласия (разделы 13, 16);
- ID, папки в облаке, переименование файлов, `description.txt` и `meta.txt` (разделы 20–23);
- реестр в БД с автогенерацией XLSX, повторные заявки (разделы 15, 25);
- мини-команды модератора в боте (раздел 27);
- автосообщения участнику и уведомления организаторам (разделы 18, 19);
- папка отклонённых через смену статуса + `reason.txt` (раздел 24).

**Ручным процессом остаётся** только то, что и в ТЗ помечено «не входит в MVP» (раздел 4): голосование, работа жюри, выдача мерча, публикация в галерею. Поля под жюри/голосование/мерч в реестре есть и заполняются вручную, бот их хранит, но процессы не автоматизирует. Веб-админку для модератора в этом проекте не делаем.
